## 开发须知

### 1. 非功能性要求

**可靠性**

- 系统在发生异常时应返回统一的错误响应格式（`code=500, msg=错误信息`），不泄露堆栈信息
- 前端请求超时设置为 60 秒，超时后提示"请求超时"
- 后端接口通过 `@Transactional` 保证数据一致性，保存/提交操作失败时自动回滚

**高性能**

- 分页查询使用 MyBatis-Plus 分页插件，避免全表扫描
- 列表页面单次查询默认返回 10 条记录，支持 10/20/50 条切换
- 响应时间：一般查询不超过 1 秒，保存/提交操作不超过 2 秒

**可维护性**

- 前后端分离架构，前端 Vue3 + Element Plus，后端 Spring Boot + MyBatis-Plus
- Entity 与 VO 分离，数据库模型变更不影响接口契约
- 统一的请求/响应拦截器，公共逻辑集中处理

**安全性**

- 前端通过 Vite 代理转发请求，避免跨域问题
- 后端使用 `@ExceptionHandler` 统一异常处理，不暴露内部错误细节
- 已作废单据不可编辑，已提交单据不可退回草稿

## 2. 功能性需求描述

### 2.1. 需求概述

#### 2.1.1. 报销单管理

- 分页查询报销单列表（支持按单号、标题、事由、公司、部门、报销人、业务类型筛选）
- 新增、编辑、提交、作废、删除报销单
- 查看报销单详情（含行程、补助、分摊信息）

#### 2.1.2. 报销单填报

- 填写基础信息：报销标题、报销人、报销部门、费用归属公司、业务类型、出差事由
- 补录行程：添加/删除/修改行程（出行人、出发城市、到达城市、日期范围、行程说明）
- 补助管理：添加行程后自动生成补助信息和补助日历，支持自定义补助金额（不可超过标准上限）
- 费用分摊：设置费用归属公司及分摊比例，比例合计必须为 100%

### 2.2. 业务全景图

```
报销单列表页
  │
  ├── 查询（分页 + 条件筛选）
  ├── 新增 ──→ 报销单详情页（编辑模式）
  ├── 编辑 ──→ 报销单详情页（编辑模式）
  ├── 查看 ──→ 报销单详情页（只读模式）
  ├── 作废 ──→ 状态变为"已作废"
  └── 删除 ──→ 物理删除（含关联数据）

报销单详情页
  │
  ├── 基础信息（报销标题、报销人、部门、公司、业务类型、出差事由）
  ├── 行程管理（添加/编辑/删除行程）
  ├── 补助信息（自动生成，可自定义金额）
  ├── 费用分摊（设置归属公司及比例）
  ├── 保存草稿 ──→ 状态 = 0（草稿）
  └── 提交 ──→ 状态 = 1（已完成），校验必填项
```

## 3. 功能详细设计

### 3.1. 功能点1：报销单分页查询

#### 3.1.1. 功能内容

列表页面展示所有报销单，支持以下操作：
- 按条件筛选：报销单号、标题、事由、费用归属公司、报销部门、报销人、业务类型
- 分页浏览：默认每页 10 条，支持 10/20/50 条切换
- 操作按钮：新增、查看、编辑、作废、删除

#### 3.1.2. 实现逻辑

**前端流程：**

1. 页面加载时调用 `GET /api/dict` 获取字典数据（公司、部门、员工、业务类型、城市）
2. 调用 `POST /api/reimbursements/page` 获取第一页数据
3. 用户输入筛选条件后点击"搜索"，重新调用分页接口
4. 点击"清除"重置筛选条件并刷新列表

**后端流程：**

1. `ReimbursementController.page()` 接收 `ReimbursementQueryDTO`
2. `ReimbursementServiceImpl.page()` 使用 MyBatis-Plus 分页插件查询
3. Mapper XML 中使用 `<where>` + `<if>` 动态拼接 SQL 条件
4. 将 Entity 转换为 `ReimbursementVO` 返回（不暴露 `version` 等内部字段）

**SQL 查询逻辑：**

```sql
SELECT * FROM fk_reim_main
WHERE reim_no LIKE '%关键字%'        -- 模糊匹配报销单号
  AND reimbursement_title LIKE '%'   -- 模糊匹配标题
  AND business_trip_reason LIKE '%'  -- 模糊匹配事由
  AND reim_company_id = ?            -- 精确匹配公司
  AND reim_department_id = ?         -- 精确匹配部门
  AND reimburser_id = ?              -- 精确匹配报销人
  AND business_type_id = ?           -- 精确匹配业务类型
ORDER BY creation_time DESC
```

#### 3.1.3. 异常处置

- 查询失败：前端捕获异常，显示"查询失败"提示
- 数据为空：列表显示空状态，分页组件显示"共0条"

### 3.2. 功能点2：新增/编辑报销单

#### 3.2.1. 功能内容

- 新增：点击"新增"按钮进入空白详情页
- 编辑：点击列表中的"编辑"图标进入已有数据的详情页
- 基础信息填写：报销标题、报销人（下拉选择）、报销部门（下拉选择）、费用归属公司（下拉选择）、业务类型（树形选择）、出差事由

#### 3.2.2. 实现逻辑

**前端流程：**

1. 新增时不传 ID，编辑时通过路由参数 `/reimbursement/{id}` 获取 ID
2. 编辑模式下调用 `GET /api/reimbursements/{id}` 加载完整数据
3. 选择报销人时自动填充报销人工号和姓名
4. 选择部门时自动填充部门编号和名称
5. 选择公司时自动填充公司编号和名称，并更新费用分摊的默认公司
6. 选择业务类型时自动填充业务类型编号和名称

**后端流程（保存/提交）：**

1. `ReimbursementController.save()` 或 `.submit()` 接收 `Reimbursement` Entity
2. `ReimbursementServiceImpl.save(item, submit)` 执行：
   - `validateHeader()` — 提交时校验必填字段
   - `rebuildSubsidies()` — 根据行程重建补助数据
   - `rebuildTotals()` — 重新计算金额合计
   - `validateTrips()` — 提交时校验行程数据
   - `validateSubsidies()` — 校验补助金额不超过标准
   - `validateAllocations()` — 提交时校验分摊比例和金额
   - 设置状态：保存=0（草稿），提交=1（已完成）
   - 新增时插入数据库，编辑时更新数据库
   - `rebuildDetails()` — 删除旧的子表数据，插入新的行程/补助/日历
3. 返回完整的 `ReimbursementVO`

#### 3.2.3. 异常处置

| 异常场景 | 错误信息 | 处理方式 |
| --- | --- | --- |
| 提交时标题为空 | 报销标题不能为空 | 返回错误，前端提示 |
| 提交时无行程 | 至少录入一条行程 | 返回错误，前端提示 |
| 分摊比例不为100% | 分摊比例合计必须为100% | 返回错误，前端提示 |
| 分摊金额不等于补助总额 | 分摊金额合计必须等于费用合计中的补助总金额 | 返回错误，前端提示 |
| 同一出行人日期重复 | 补录行程中同一人员日期范围不可重复 | 返回错误，前端提示 |
| 已作废单据编辑 | 已作废单据不可编辑 | 返回错误，前端提示 |
| 已提交单据退回草稿 | 已提交单据不可退回草稿 | 返回错误，前端提示 |

### 3.3. 功能点3：行程管理

#### 3.3.1. 功能内容

- 添加行程：选择出行人、出发城市、到达城市、出发/到达日期、行程说明
- 编辑行程：修改已有行程信息
- 删除行程：移除行程及其关联的补助数据
- 行程校验：日期不可倒挂、不可晚于当前日期、同一出行人日期不可重复

#### 3.3.2. 实现逻辑

**前端流程：**

1. 点击"添加行程"弹出行程表单弹窗
2. 选择出行人后自动填充工号和姓名
3. 选择城市后自动填充城市编号
4. 日期范围使用 `el-date-picker` 的 `daterange` 模式
5. 保存行程后自动调用 `saveDraft(false)` 保存草稿
6. 删除行程后同步删除关联的补助数据

**后端流程：**

1. `validateTrips()` 校验行程数据：
   - 出行人、出发城市、到达城市、日期、行程说明不能为空
   - 到达日期不可早于出发日期
   - 到达日期不可晚于当前日期
   - 同一出行人的行程日期范围不可重叠

#### 3.3.3. 异常处置

- 出行人为空：提示"出行人不能为空"
- 日期倒挂：提示"到达日期不可早于出发日期"
- 日期超前：提示"到达日期不可晚于当前日期"
- 日期重复：提示"补录行程中同一人员日期范围不可重复"

---

### 3.4. 功能点4：补助管理

#### 3.4.1. 功能内容

- 添加行程后自动生成补助信息（含补助日历）
- 补助日历默认全选标准金额
- 支持自定义补助金额（不可超过标准上限）
- 补助标准根据城市类型决定：一线 100/天，二线 80/天，其他 50/天

#### 3.4.2. 实现逻辑

**自动生成逻辑：**

1. 添加行程后，`rebuildSubsidies()` 根据行程自动生成补助数据
2. 每个行程对应一条补助记录
3. 补助天数 = 结束日期 - 开始日期 + 1
4. `rebuildCalendar()` 为每天生成一条日历明细：
   - 默认选中餐费、交通、通讯三项
   - 餐费标准根据到达城市类型决定
   - 交通标准默认 40.00，通讯标准默认 40.00

**自定义金额逻辑：**

1. 点击补助行的编辑按钮，打开日历明细弹窗
2. 可取消选中某项补助（不勾选则金额为 0）
3. 可修改金额，但不可超过标准上限
4. 保存后重新计算补助金额合计

**校验规则：**

```java
// 未选中时金额必须为 0
if (!selected && amount != 0) throw "XX补助未选中时金额必须为0";
// 选中时金额需为正数且不可大于标准
if (selected && (amount < 0 || amount > standard)) throw "XX补助金额需为正数且不可大于标准金额";
```

#### 3.4.3. 异常处置

- 金额超限：提示"XX补助金额需为正数且不可大于标准金额"
- 未选中但有金额：提示"XX补助未选中时金额必须为0"

---

### 3.5. 功能点5：费用分摊

#### 3.5.1. 功能内容

- 设置费用归属公司及分摊比例
- 支持多行分摊，比例合计必须为 100%
- 分摊金额合计必须等于补助总金额
- 至少保留一条分摊信息

#### 3.5.2. 实现逻辑

**前端逻辑：**

1. 新增报销单时默认生成一行分摊，归属公司为主表的费用归属公司
2. 点击"添加"新增分摊行
3. 修改第 2+ 行的分摊比例时，自动计算第 1 行比例 = 100% - Σ(第 2+ 行比例)
4. 若 Σ(第 2+ 行) > 100%，清空输入框
5. 分摊金额 = 补助总金额 × 分摊比例
6. 删除分摊行时，若只有 1 行则提示"至少保留一条分摊信息"

**后端校验：**

```java
// 提交时校验
if (submit && ratio.compareTo(BigDecimal.ONE) != 0)
    throw "分摊比例合计必须为100%";
if (submit && amount.compareTo(item.getSubsidyTotal()) != 0)
    throw "分摊金额合计必须等于费用合计中的补助总金额";
```

#### 3.5.3. 异常处置

- 比例不为 100%：提示"分摊比例合计必须为100%"
- 金额不匹配：提示"分摊金额合计必须等于费用合计中的补助总金额"
- 费用归属为空：提示"费用归属不能为空"

---

### 3.6. 功能点6：作废/删除报销单

#### 3.6.1. 功能内容

- 作废：将单据状态改为"已作废"，不可重复作废
- 删除：物理删除单据及所有关联数据（行程、补助、日历），不论状态均可删除

#### 3.6.2. 实现逻辑

**作废流程：**

1. 前端弹出确认框："确定作废当前单据吗？"
2. 调用 `POST /api/reimbursements/{id}/void`
3. 后端校验：单据不存在 → 报错；已作废 → 报错
4. 更新 `status = 2`

**删除流程：**

1. 前端弹出确认框："删除后不可恢复，确定删除当前单据吗？"
2. 调用 `POST /api/reimbursements/{id}/delete`
3. 后端按顺序删除：日历明细 → 补助 → 行程 → 主表
4. 使用 `@Transactional` 保证原子性

#### 3.6.3. 异常处置

- 单据不存在：提示"单据不存在"
- 重复作废：提示"单据已作废，不可重复作废"

### 4.3. 与前端的交互

**前端组件与后端接口对应关系：**

| 前端组件 | 调用接口 | 说明 |
| --- | --- | --- |
| `ReimbursementList.vue` | `POST /api/reimbursements/page` | 分页查询列表 |
| `ReimbursementList.vue` | `POST /api/reimbursements/{id}/void` | 作废报销单 |
| `ReimbursementList.vue` | `POST /api/reimbursements/{id}/delete` | 删除报销单 |
| `ReimbursementDetail.vue` | `GET /api/reimbursements/{id}` | 加载详情 |
| `ReimbursementDetail.vue` | `POST /api/reimbursements/save` | 保存草稿 |
| `ReimbursementDetail.vue` | `POST /api/reimbursements/submit` | 提交报销单 |
| 两个页面共用 | `GET /api/dict` | 获取字典数据 |

**前端请求工具（request.ts）：**

- 基于 axios 封装，baseURL = `/api`
- 请求拦截：自动添加 loading 效果

## 5. 关键技术点

### 5.1. 并发编程

本系统为内部管理系统，用户量小，不涉及高并发场景。暂不使用并发编程。

### 5.2. 事务控制

#### 5.2.1 数据库局部事务

以下操作使用 `@Transactional` 注解保证数据一致性：

| 方法 | 事务范围 | 说明 |
| --- | --- | --- |
| `save()` | 主表 + 行程 + 补助 + 日历 + 分摊 | 保存/提交报销单 |
| `delete()` | 主表 + 行程 + 补助 + 日历 | 删除报销单 |
| `rebuildDetails()` | 行程 + 补助 + 日历 | 重建子表数据 |

#### 5.2.2 分布式事务

本系统为单体应用，不涉及分布式事务。

### 5.3. Job使用

本系统不涉及定时任务。

### 5.4.权限控制

当前版本未实现权限控制，所有用户可访问所有数据。后续可扩展：
- 数据权限：按公司隔离数据
- 功能权限：按角色控制增删改查

### 5.5. Redis使用

当前版本未使用 Redis。后续可扩展：
- 字典数据缓存（公司、部门、员工等变动频率低）
- 用户会话管理

### 5.6. 敏感信息处理

当前版本不涉及敏感信息。后续如需存储身份证、银行卡等信息，需加密存储、脱敏显示。

### 5.7. 错误码使用

| 错误码 | 含义 | 使用场景 |
| --- | --- | --- |
| 200 | 成功 | 所有成功响应 |
| 500 | 业务异常 | 参数校验失败、业务规则违反 |

错误信息通过 `Result.fail(msg)` 返回，前端直接弹出 `msg` 提示用户。

### 5.8. 异动日志

当前版本未实现异动日志。后续可扩展：
- 记录报销单的创建、修改、提交、作废操作
- 记录操作人、操作时间、操作内容

### 5.9. 大数据量问题

- 分页查询使用 MyBatis-Plus 分页插件，避免全表加载
- 子表数据（行程、补助、日历）按需加载，列表接口不返回
- 费用分摊以 JSON 形式存储在主表，避免额外的关联查询
- 数据量预估：年均数千条报销单，无性能瓶颈

### 5.10. 缓存的使用

当前版本未使用缓存。字典数据每次请求都从数据库查询，数据量小，性能可接受。

### 5.11. MQ的使用

当前版本不涉及消息队列。

### 5.12. 重试机制使用

当前版本不涉及远程调用，无需重试机制。
